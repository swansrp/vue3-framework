# Dataset配置管理 - 功能特性说明

## 核心特性

### 1. 智能SQL解析

系统能够自动解析标准的SELECT SQL语句,提取表结构和列信息:

- **表解析**:自动识别表名、别名、JOIN类型和JOIN条件
- **列解析**:自动识别列的SQL表达式、别名、是否使用聚合函数
- **JOIN支持**:支持INNER、LEFT、RIGHT、FULL JOIN
- **聚合函数**:自动识别COUNT、SUM、AVG、MAX、MIN等聚合函数

### 2. Dataset名称规范

**自动后缀添加**:
- 用户输入:`orderSummary` → 系统保存:`orderSummaryDataset`
- 前端validator强制校验Dataset名称必须以"Dataset"结尾
- 编辑时自动移除后缀供用户修改,保存时再添加回来

**好处**:
- 统一命名规范,避免混乱
- 一眼识别Dataset配置
- 防止与其他表名冲突

### 3. 三标签页SQL管理

**编辑SQL标签页**:
- Ace Editor代码编辑器,提供SQL语法高亮
- 一键格式化SQL功能
- 实时预览解析结果
- SQL示例库(简单查询、多表JOIN、聚合函数)

**结构说明标签页**:
- 可视化展示表配置(表名、别名、JOIN信息)
- 可视化展示列配置(SQL表达式、别名、是否聚合等)
- JOIN条件以代码块形式清晰展示

**压缩SQL标签页**:
- 单行紧凑格式,方便复制
- 适合粘贴到其他工具使用
- 一键复制到剪贴板

### 4. 列配置管理

**自动生成**:
- SQL解析时自动创建列配置
- 自动识别聚合字段
- 自动设置初始显示顺序

**手动调整**:
- 添加新列或修改现有列
- 上移/下移调整显示顺序(失败时自动重新加载)
- 隐藏不需要显示的列
- 为列添加备注说明

**别名处理**:
- 自动去除别名两侧的单引号
- 如:`'orderNo'` 显示为 `orderNo`

### 5. 数据结构设计

**三表关联**:
```
sys_dataset (主表)
  ├─ sys_dataset_table (表配置,1对多)
  └─ sys_dataset_column (列配置,1对多)
```

**级联操作**:
- 删除Dataset时自动级联删除tables和columns
- 更新Dataset时自动更新关联配置

### 6. 用户体验优化

**搜索功能**:
- 支持Dataset名称的模糊搜索
- 支持备注的模糊搜索
- 使用OR条件,满足任一即可

**单选联动**:
- 点击行任意位置选中/取消选中
- 点击Dataset名称打开编辑弹窗
- 点击按钮或链接不触发选中

**自动刷新**:
- 保存后自动刷新列表
- 列配置抽屉关闭后自动刷新主列表
- 删除后自动清空选中状态(如果删除的是当前选中项)

**性能优化**:
- 并发加载Dataset、Tables、Columns三张表数据
- 减少等待时间,提升加载速度

## 应用场景

### 场景1:复杂报表查询

创建涉及多表JOIN和聚合函数的数据查询配置,无需编写代码即可实现报表查询功能。

**示例**:订单汇总报表
```sql
SELECT 
  o.customer_id AS customerId,
  c.name AS customerName,
  COUNT(o.id) AS orderCount,
  SUM(o.amount) AS totalAmount
FROM t_order o
LEFT JOIN t_customer c ON o.customer_id = c.id
GROUP BY o.customer_id, c.name
```

### 场景2:数据分析视图

快速创建数据分析所需的各种统计视图,支持灵活的条件筛选。

**示例**:月度销售统计
```sql
SELECT 
  DATE_FORMAT(created_at, '%Y-%m') AS month,
  COUNT(id) AS orderCount,
  SUM(amount) AS totalAmount,
  AVG(amount) AS avgAmount
FROM t_order
GROUP BY DATE_FORMAT(created_at, '%Y-%m')
```

### 场景3:业务数据导出

配置好Dataset后,业务人员可以自助查询和导出所需数据,减少开发工作量。

## 最佳实践

### SQL编写

1. **必须使用表别名**:多表查询时每个表都要指定别名
2. **建议使用列别名**:使用AS关键字为列指定有意义的别名
3. **先测试后使用**:复杂SQL建议先在数据库中测试通过
4. **避免SELECT ***:明确指定需要的列

### Dataset命名

1. **语义化**:使用有意义的名称,如`orderSummary`、`customerList`
2. **驼峰命名**:使用驼峰命名法或下划线命名法
3. **简洁明了**:保持名称简短但描述准确

### 列配置

1. **合理排序**:重要字段排在前面
2. **隐藏技术字段**:如ID、创建时间等技术字段可以隐藏
3. **添加备注**:为业务字段添加清晰的备注说明
4. **正确标记**:准确标记聚合字段

### 查询优化

1. **创建索引**:为常用查询条件的字段创建索引
2. **避免全表扫描**:尽量使用索引字段作为查询条件
3. **合理分页**:大数据量查询使用分页
4. **物化视图**:高频复杂查询考虑使用物化视图

## 常见问题

### Q:修改Dataset名称后会影响已有查询吗?

A:会的。Dataset名称是系统识别的唯一标识,修改后需要更新相关的查询配置。建议:
- 创建时就使用最终名称
- 如需修改,确保相关查询一并更新

### Q:可以在SQL中使用子查询吗?

A:可以,但建议:
- 简单场景优先使用JOIN
- 复杂子查询先在数据库测试
- 注意子查询的性能影响

### Q:聚合字段的查询条件为什么要放在HAVING中?

A:这是SQL语法要求:
- WHERE用于过滤原始数据
- HAVING用于过滤聚合后的结果
- 系统会根据字段的isAggregate属性自动选择

### Q:删除列配置会影响已保存的查询条件吗?

A:会的。删除列后:
- 该列不再显示在查询界面
- 之前保存的包含该列的查询条件会失效
- 建议改用"隐藏"而非"删除"
